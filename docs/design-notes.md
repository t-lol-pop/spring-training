# 設計ノート — 各クラスの役割と設計理由（2026-07-12時点）

このドキュメントは、これまでに作成したクラス一つひとつについて「何をするクラスか」「なぜその設計にしたか」を初心者向けに解説するものです。クラス同士の関係を図で見たい場合は [class-diagram.md](./class-diagram.md) を参照してください。

## `com.training.issuing.domain` パッケージ

決済システムの中心となる「業務ルール」を表現するクラス群です。Spring Bootや外部フレームワークに一切依存しない、素のJavaクラスだけで構成されています。

### `Money`

**役割**: 金額（日本円）を表す。

**設計理由**:
- 金額を`long`のような単なる数値として扱わず、専用のクラスで包んでいます（Primitive Obsessionの回避）。こうすることで、「会員ID」や「カード枚数」など他の数値と間違えて渡してしまう事故を防げます。
- `amount`フィールドは`private final`にし、一度作ったら値が変わらない「不変オブジェクト」にしています。`plus`/`minus`は自分自身を書き換えるのではなく、常に**新しい`Money`インスタンスを作って返す**設計です。これにより、複数の場所から同時に使われても壊れない（スレッドセーフ）、いつ・誰が値を変えたか追跡しやすい（監査ログ向き）というメリットがあります。
- 内部表現は`long`（円の整数）を採用しました。日本円には小数がないため`BigDecimal`は不要と判断し、また`int`では最大約21億円までしか扱えずオーバーフローの危険があるため`long`にしています。
- 残高不足の減算は`InsufficientBalanceException`という独自の例外を投げます（後述）。

### `Member`

**役割**: 会員（お金の持ち主・カード利用者）を表す。

**設計理由**:
- 会員ID・氏名・入会日・状態（`Status`）を持ちます。
- 状態は`enum Status（ACTIVE, SUSPENDED, WITHDRAWN）`で表現しています。`boolean isActive`のようなフラグの組み合わせにしなかったのは、将来「一時停止理由」のような状態が増えても、enumの選択肢を増やすだけで済み、変更に強いためです。
- 状態を変更する手段として、`suspend()`・`withdraw()`という**業務的な意味を持つメソッド**を用意しました。汎用的な`setStatus(Status s)`のようなsetterにしなかったのは、「なぜ状態が変わったか」をメソッド名自体が語るようにするためです。
- `Member`同士が「同じ会員かどうか」を比較する`equals`/`hashCode`は、**会員IDだけ**を基準にしています。氏名や入会日が違っていても、IDが同じなら同一人物とみなします。これは「エンティティ（同一性で識別されるもの）」の考え方で、値そのもので比較する`Money`（値オブジェクト）とは意図的に基準を変えています。

### `InsufficientBalanceException`

**役割**: `Money`の減算で残高が不足したときに投げる例外。

**設計理由**:
- `RuntimeException`（非チェック例外）を継承しています。残高不足は「呼び出し元に必ず`try-catch`を強制したいAPI境界のエラー」というよりは「業務上ときどき起こりうるエラー」なので、チェック例外にはせず、必要な場所（後述の`@RestControllerAdvice`など）で一元的に処理する設計にしました。
- クラス名は当初`MinusException`という実装都合の名前でしたが、「呼び出し側が本当に知りたいのは何か（残高不足という業務的な意味）」という観点でリファクタリングし、今の名前にしています。

## `com.training.issuing.greeting` パッケージ

Spring BootのDI（依存性注入）の仕組みを学ぶための、練習用の小さな機能群です。

### `Greeter`（インターフェース）

**役割**: 「名前を渡すと挨拶メッセージを返す」という振る舞いの型を定義する。

**設計理由**:
- 実装クラス（`SimpleGreeter`など）ではなく、この**インターフェースに対して依存する**ことで、呼び出し側（`GreetingRunner`）は「具体的にどの挨拶をするか」を知らなくてよくなります。これにより、本物の実装を変更せずにテスト用の偽物と差し替えられる（疎結合・テスト容易性）というメリットが生まれます。

### `SimpleGreeter` / `FormalGreeter`

**役割**: `Greeter`の具体的な実装（カジュアルな挨拶・丁寧な挨拶）。

**設計理由**:
- `@Component`を付けてSpringにBean（管理対象のオブジェクト）として登録しています。`@Service`ではなく`@Component`にしたのは、これらが「業務ドメインの知識を持たない汎用的な部品」だからです。
- 実装クラスが2つ存在するため、Springがどちらを注入すべきか迷うことがあります。それを解消するために`SimpleGreeter`にだけ`@Primary`を付け、「候補が複数あるときのデフォルト」として指定しています。

### `GreetingRunner`

**役割**: アプリ起動時に一度だけ、挨拶メッセージをコンソールに出力する。

**設計理由**:
- `Greeter`を**コンストラクタインジェクション**で受け取っています。フィールドインジェクション（`@Autowired`をフィールドに直接付ける方法）にしなかったのは、コンストラクタインジェクションの方が「Springを起動しない素のJavaテストが書ける」「フィールドを`final`にできる（不変性を保てる）」という利点があるためです。
- `CommandLineRunner`というSpring Bootの仕組みを使うことで、「アプリ起動完了直後に実行したい処理」を自然に表現しています。

## `com.training.issuing.point` パッケージ

外部設定ファイル（`application.yml`）から値を読み込む仕組みを学ぶための機能群です。

### `PointProperties`

**役割**: `application.yml`の`issuing.point.rate`という設定値を、型安全な形で保持する。

**設計理由**:
- Javaの`record`を使っています。「設定値を保持するだけの、変更されないデータの入れ物」という役割にちょうど合うため、不変性を手軽に実現できる`record`を選びました。
- `@ConfigurationProperties`を使うことで、`@Value("${...}")`のように文字列のキー名を個別にあちこちに書くのではなく、関連する設定値を1つのクラスにまとめられます。将来設定項目が増えても、見通しを保ちやすくなります。
- `@Validated`+`@DecimalMin`で「マイナスの付与率」のような不正な設定値を、アプリ起動時に検知できるようにしています。設定ミスに気づかないまま起動してしまう事故を防ぎます。

### `PointCalculator`

**役割**: 購入金額とポイント付与率から、実際のポイント数を計算する。

**設計理由**:
- `PointProperties`をコンストラクタインジェクションで受け取ります。テストのときは実際の`application.yml`を読み込まず、任意の付与率を持つ`PointProperties`を直接渡してテストできます。

### `PointRunner`

**役割**: 起動時にサンプル計算を実行し、設定値が正しく反映されているかを目視確認するための、動作確認専用クラス。

**設計理由**:
- 本番のロジックではなく、あくまで「プロファイルを切り替えたときに値が変わることを確認する」ための一時的な道具です。将来、REST APIなどの正式な呼び出し口ができれば、役目を終えて削除される想定です。

## `com.training.issuing.member` パッケージ

会員登録・照会のREST APIと、そのデータをDBに保存する仕組み一式です。「コントローラー → サービス → リポジトリ」という、実務でよくあるレイヤードアーキテクチャの構造になっています。

### `MemberController`

**役割**: HTTPリクエスト（`POST /members`、`GET /members/{id}`）を受け取り、レスポンスを返す。

**設計理由**:
- 業務ロジック（ID採番や「見つからない場合の判定」など）はここには書かず、`MemberService`に処理を委譲しています。コントローラーの役割は「HTTPの入出力の変換」に限定するためです。
- `POST`成功時は`201 Created`と、作成されたリソースを指す`Location`ヘッダーを返します。「新しいリソースが作られた」ことを正しく表現するためです。

### `MemberRegisterRequest` / `MemberResponse` / `ErrorResponse`（DTO）

**役割**: それぞれ「登録リクエストの形」「レスポンスの形」「エラーレスポンスの形」を表す。

**設計理由**:
- `Member`ドメインオブジェクトをそのままAPIの入出力に使いませんでした。理由は、ドメインモデルとAPIの外部仕様は「変化する理由」が違うためです。`Member`に業務都合でフィールドが増えても、API仕様がそれに引きずられて変わってしまうのは望ましくありません。また、意図しない情報（将来追加されるかもしれない内部管理用の情報など）が外部に漏れるリスクも防げます。
- `id`と`registeredDay`はリクエストに含めていません。会員IDはサーバー側で`UUID`により採番し、入会日はサーバー側の現在日時を使います。クライアントに登録日時を偽って指定させないためです。

### `MemberNotFoundException`

**役割**: 指定したIDの会員が存在しないことを表す例外。

**設計理由**: `InsufficientBalanceException`と同様、`RuntimeException`を継承した非チェック例外です。

### `MemberExceptionHandler`

**役割**: `MemberNotFoundException`（→`404`）、入力検証エラー（→`400`）を一箇所でまとめて処理する。

**設計理由**:
- `@RestControllerAdvice`を使うことで、各コントローラーメソッドに個別の`try-catch`を書かずに済みます。これにより、エラーレスポンスの形式がアプリ全体で統一され、メンテナンスもしやすくなります。

### `MemberService`

**役割**: 会員登録・照会という「ユースケース」を実現する。

**設計理由**:
- `@Service`を付けています。`Greeter`の実装（`@Component`）とは違い、こちらは「会員登録」という明確な業務ロジックを持つため、意図を示す`@Service`を選びました。
- ID採番（`UUID`生成）や「見つからなければ例外を投げる」という判断はここに集約し、コントローラーやリポジトリには持たせていません。

### `MemberRepository`

**役割**: `Member`の保存・取得を担う。

**設計理由**:
- 内部の実装（最初はインメモリの`Map`、現在はJPA経由でのDB保存）を、外部（`MemberService`）から隠しています。`save`/`findById`という公開APIは変えずに内部実装だけ差し替えたので、**リポジトリの実装をJPAに変更しても`MemberService`や`MemberController`は一切修正不要**でした。これが「関心の分離」の実務的なメリットです。

### `MemberJpaEntity`

**役割**: JPA（Hibernate）がDBとやり取りするための、永続化専用のクラス。

**設計理由**:
- `Member`ドメインクラスに直接`@Entity`を付けませんでした。JPAは「引数なしコンストラクタ」と「リフレクションによるフィールドへの直接代入」を要求しますが、これは`Member`が大事にしてきた「コンストラクタで必ず正しい状態を作る」「カプセル化」という設計方針と相性が悪いためです。`MemberJpaEntity`を別クラスにすることで、`Member`をこうした技術的な制約から解放しています。
- `fromDomain`/`toDomain`メソッドで、`Member`との相互変換を行います。

### `MemberJpaRepository`

**役割**: Spring Data JPAの仕組みを使い、DBへのCRUD操作を自動生成してもらう。

**設計理由**: インターフェースを定義するだけで、Spring Data JPAが実装を自動的に用意してくれます。SQLを手で書く必要がありません。

## `com.training.issuing.IssuingApplication`

**役割**: アプリ全体の起動地点。

**設計理由**:
- `@SpringBootApplication`が持つ`@ComponentScan`により、同じパッケージ以下（`greeting`、`point`、`member`など）にある`@Component`/`@Service`/`@Repository`付きのクラスを自動的に見つけてBean登録できます。そのためこのクラスは、プロジェクトの一番上の階層（`com.training.issuing`）に置いています。
- `@EnableConfigurationProperties(PointProperties.class)`を追加し、`record`で書いた`PointProperties`をBeanとして明示的に登録しています。

## 全体を通じての設計の一貫性

- **不変性**: `Money`、`PointProperties`、各種DTOなど、「値」を表すクラスは基本的に不変にしています。
- **カプセル化**: フィールドは`private`にし、外部から不正な状態変更ができないようにしています。
- **関心の分離**: ドメイン（業務ルール）・DTO（API外部仕様）・JPAエンティティ（永続化）を、それぞれ別クラスとして分離しています。同じ「会員」という概念でも、目的が違えば別のクラスとして表現する、という考え方が一貫しています。
- **依存性注入（DI）**: すべてのクラス間の依存は、`new`で直接生成せず、Springのコンストラクタインジェクションを通じて解決しています。
