# Level2-1: DI/IoCの基礎 — 会員向け挨拶メッセージを起動時に表示する

- 提示日: 2026-07-05
- 状態: 完了（2026-07-05）

## 学習テーマ
Spring BootのDI（依存性注入）/ IoC（制御の反転）の仕組みを、最小構成のアプリで体感する。

## ゴール
Springコンテナが「オブジェクトの生成・注入」を肩代わりしてくれる仕組みを理解し、`@Component`のようなステレオタイプアノテーションと、コンストラクタインジェクションを使えるようになる。

## 要件

1. `Greeter` というインターフェースを作る。`String greet(String memberName)` のようなメソッドを1つ持つ。
2. `Greeter` を実装するクラス（例: `SimpleGreeter`）を作り、Beanとして登録する。
3. アプリ起動時に一度だけ実行される `CommandLineRunner` を実装したクラス（例: `GreetingRunner`）を作り、コンストラクタで `Greeter` を受け取り、起動時にコンソールへ挨拶メッセージを出力する。
4. `./gradlew bootRun` で実際にアプリを起動し、コンソールに挨拶メッセージが表示されることを確認する。

配置場所の目安: `src/main/java/com/training/issuing/greeting/` のような新しいパッケージを切ってよい（`domain`パッケージとは分離する）。

## 制約
- `new SimpleGreeter()` のように自分で明示的にインスタンス化してはいけない。Springに生成・注入させること。
- フィールドインジェクション（`@Autowired`をフィールドに直接付ける方法）ではなく、コンストラクタインジェクションを使うこと。

## 設計上の論点

1. なぜ`new SimpleGreeter()`と自分で書かずに、Springにインスタンス化させるのか？そのメリットは何か？（テスト容易性、疎結合という観点で考えてみてください）
2. `@Component`、`@Service`は技術的にはほぼ同じ働きをしますが、なぜ使い分けるのか？
3. コンストラクタインジェクションとフィールドインジェクション、それぞれのメリット・デメリットは？実務ではどちらが推奨されるか？
4. もし`Greeter`の実装クラスが2つ（例: `SimpleGreeter`と`FormalGreeter`）存在したら、Springはどちらを注入すればよいか判断できると思いますか？何が起きるでしょうか？（実際に試してみるのも良い実験です）

## 実装ヒント（答えにならない程度に）
- Springに「このクラスはBeanとして管理してほしい」と伝えるアノテーションが必要です
- `CommandLineRunner`はSpring Bootが提供するインターフェースで、`run(String... args)`メソッドを実装すると起動時に自動実行されます
- コンストラクタインジェクションでは、Spring Boot 2.x以降なら`@Autowired`アノテーション自体を省略できる場合があります（コンストラクタが1つだけの場合）

## テスト観点
- 単体テスト: `SimpleGreeter`単体の`greet()`の戻り値を検証する（Springを起動せず、素の単体テストとして書けるはず。これがDIのメリットの一つです）
- 動作確認: `./gradlew bootRun`でアプリを起動し、ログに正しいメッセージが出力されるか目視確認する

## レビュー観点
- インターフェースと実装クラスの分離ができているか
- コンストラクタインジェクションが正しく使えているか
- Beanのアノテーション選択が適切か
- テストがSpringコンテナを起動せずに書けているか（unit testとintegration testの違いを意識できているか）

## 学ぶべき設計原則
DI（依存性注入）、IoC（制御の反転）、疎結合、インターフェースに対するプログラミング

## レビューログ

**実装**: 要件1〜4はユーザーの依頼により、メンター（Claude）が直接実装（`Greeter`, `SimpleGreeter`, `GreetingRunner`）。実装後、設計論点についてユーザーに問答形式で確認した。

**設計論点の理解度**:
1. 「なぜSpringに生成させるのか」の初回の回答は「ベストプラクットだから」という結論のみで、理由の言語化ができていなかった。具体例（テスト時にフェイクの`Greeter`と差し替えられるか）で問い直したところ、「実装クラスのコードを変えなくてもテストができる」と自分の言葉で説明できるようになった。疎結合・依存性逆転の考え方の入り口として言及（Level6で正式に扱う）
2. `@Component` vs `@Service`の違いを知らなかったため、技術的な違いはほぼ無く意味的な違い（コードを読む人に意図を伝えるためのもの）であることを説明。その上で`SimpleGreeter`は業務ドメインの知識を持たないので`@Component`のままで妥当、と正しく判断できた
3. `@Autowired`（フィールドインジェクション）の仕組み（リフレクションで`private`フィールドに後から代入）を知らなかったため説明。その後「コンストラクタインジェクションの方が素のJavaテストを書きやすい」「`final`にできるのもコンストラクタインジェクションだけ」と正しく結論づけられた
4. 複数実装がある場合の挙動について「両方を順番に注入する」という誤った予想をしたが、実際に`FormalGreeter`を追加して`bootRun`を実行し、`UnsatisfiedDependencyException`（起動失敗）になることを実地で確認。「曖昧な依存は起動時エラーとして明示的に検出される」という設計思想を体感できた
5. `@Primary`をユーザーの依頼によりメンターが実装し、複数候補があっても優先Beanが解決されることを確認。`FormalGreeter`は`@Primary`/`@Qualifier`の実例として教材にそのまま残す方針

**つまずいた点（Spring Boot自体の前提知識）**:
- `CommandLineRunner`、`@Autowired`の仕組みなど、Spring Boot固有の概念を知らない状態だったため、これらは前提知識として直接説明した（設計判断ではなく事実の説明であるため）

**環境面の問題**:
- GradleデーモンがHomebrewのJava25で動作し、Spring Bootのgradleプラグイン読み込み時に`Unsupported class file major version 69`エラーが発生。`gradle.properties`で`org.gradle.java.home`をsdkman管理のJava21に固定して解消（詳細は[SETUP.md](../SETUP.md)参照）

**テスト**: `SimpleGreeterTest`を作成（ユーザーの依頼によりメンターが実装）。Springコンテナを起動せず、`new SimpleGreeter()`で素のJUnitテストとして書けることを確認
