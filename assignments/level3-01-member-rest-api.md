# Level3-1: 会員登録・照会のREST API

- 提示日: 2026-07-05
- 状態: 完了（2026-07-06）

## 学習テーマ
RESTfulなAPI設計の基礎 — リソース設計、DTOによるドメインモデルの隠蔽、入力検証、例外ハンドリングの一元化。

## ゴール
Level1で作った`Member`ドメインモデルを土台に、会員の登録・照会ができるシンプルなREST APIを実装する。ドメインモデルをそのままAPIの入出力に使わず、DTO（リクエスト/レスポンス専用のクラス）を介して公開する設計を体験する。

## 要件

1. `POST /members` で会員を新規登録できるエンドポイントを作る。リクエストボディには氏名・入会日などを含める。
2. `GET /members/{id}` で、登録した会員の情報を取得できるエンドポイントを作る。
3. リクエストの入力検証（例: 氏名が空でないこと）をBean Validationで行う。検証エラー時は`400 Bad Request`を返す。
4. 存在しない`id`を`GET /members/{id}`で指定した場合は`404 Not Found`を返す。
5. 上記3・4のような例外処理を、各コントローラーメソッドに個別の`try-catch`を書くのではなく、`@ControllerAdvice`で一元的にハンドリングする。
6. 会員データの永続化はまだDBを使わず、メモリ上（例: `Map`を使ったリポジトリクラス）で保持してよい（DBはLevel4で導入します）。

配置場所の目安: `src/main/java/com/training/issuing/member/` に、コントローラー・DTO・リポジトリなどをまとめてよい（パッケージ構成は自分で考えてみてください）。

## 制約
- `Member`ドメインオブジェクトを、そのままリクエストボディやレスポンスボディとして使わないこと。DTOクラスを用意すること。
- コントローラーに業務ロジック（`Member`の生成やバリデーション以外の判断）を書きすぎないこと。

## 設計上の論点

1. なぜ`Member`エンティティをそのままAPIの入出力に使わず、DTOを用意する必要があるのでしょうか？（考えられるリスクを挙げてみてください。例えば、`Member`にフィールドが増えたときにAPIの外部仕様にどう影響するか、なども考えてみてください）
2. URL設計について、`/members`ではなく`/getMember`や`/createMember`のような動詞を使ったURLにしないのはなぜでしょうか？RESTの考え方に基づいて説明してみてください。
3. 入力検証は「コントローラー層」「ドメイン層（`Member`自身）」のどちらで行うべきでしょうか？両方で行うべき場合、それぞれの役割の違いは何だと思いますか？
4. `@ControllerAdvice`を使わずに、各コントローラーメソッドで個別に`try-catch`していたら、どんな問題が起きると思いますか？
5. `POST /members`が成功した場合、`200 OK`と`201 Created`のどちらを返すべきだと思いますか？その違いは何でしょうか？

## 実装ヒント（答えにならない程度に）
- `@RestController`、`@RequestMapping`、`@PostMapping`、`@GetMapping`といったアノテーションを使います
- リクエストDTOのフィールドには`@NotBlank`のようなBean Validationアノテーションを付け、コントローラーの引数には`@Valid`を付けます
- `@ControllerAdvice`と`@ExceptionHandler`を組み合わせることで、特定の例外が投げられたときのレスポンスを一箇所にまとめられます
- メモリ上の保存には、スレッドセーフな`ConcurrentHashMap`などが候補になります

## テスト観点
- 正常系: 会員を登録し、そのIDで照会すると登録した内容が返ってくる
- 異常系: 氏名が空のリクエストを送ると`400`が返る
- 異常系: 存在しないIDで照会すると`404`が返る

テストの実装方法は、`@SpringBootTest`+`MockMvc`（コントローラーをHTTPリクエストに近い形でテストする方法）を使うことになりますが、詳しいやり方は着手時に相談してください。

## レビュー観点
- ドメインモデルとDTOの分離ができているか
- URL設計・HTTPメソッド・ステータスコードがRESTfulな慣習に沿っているか
- 例外ハンドリングが一元化されているか
- コントローラーに業務ロジックが漏れ出していないか

## 学ぶべき設計原則
リソース指向設計、DTOパターン、関心の分離（Controller/Service/Domainの役割分担）、一元的な例外ハンドリング

## レビューログ

**実装**: ユーザーの依頼により、要件1〜6・追加のLocationヘッダー対応・テストまで含めてメンター（Claude）が直接実装。実装後、設計論点についてユーザーに問答形式で確認した。

**実装物** (`com.training.issuing.member`パッケージ):
- `MemberRegisterRequest`（リクエストDTO、`@NotBlank`）/ `MemberResponse`（レスポンスDTO、`Member`から変換する`from`ファクトリメソッド）
- `MemberRepository`（`@Repository`、`ConcurrentHashMap`によるインメモリ保存）
- `MemberService`（`@Service`、UUIDによるID採番、登録・照会のユースケース）
- `MemberController`（`POST /members`→`201`+`Location`ヘッダー、`GET /members/{id}`）
- `MemberNotFoundException` / `ErrorResponse` / `MemberExceptionHandler`（`@RestControllerAdvice`で400/404を一元処理）
- `MemberControllerTest`（`@SpringBootTest`+`MockMvc`で正常系・バリデーションエラー・404を検証、3テストとも成功）
- 副作用として`Member`ドメインクラスにgetterを追加（DTO変換に必要だったため）

**設計判断**:
- `id`（UUID）と`registeredDay`（`LocalDate.now()`）はリクエストに含めず、サーバー側で決定する設計にした。クライアントに登録日時を偽らせないため
- `MemberController`は入出力の変換に専念させ、業務ロジック（ID採番、Not Found判定）は`MemberService`に寄せた

**設計論点の理解度**:
1. 「そもそもDTOとは何か」から質問があったため説明。その後「業務ロジックとAPI外部仕様が一体化すると、フィールド追加が即座にAPI仕様変更になり外部クライアントに影響が及ぶ」と正しく説明できた。メンターから「意図しない情報漏洩のリスク」も補足
2. URL設計について「HTTPメソッド自体が動詞の役割を持つため、URLに動詞を含める必要がない」と正確に回答。安全性・冪等性の概念も補足
3. 当初「コントローラー層の方がレスポンスが早い」という誤った理由付けをしていたが、`Money.minus()`の例（バッチ処理など別の呼び出し元からも呼ばれた場合を想像させる思考実験）を通じて、「ドメイン層の検証はどの呼び出し元からも守られるべき不変条件、コントローラー層の検証は入り口ごとの形式チェック」という正しい区別に到達した
4. `@ControllerAdvice`が無い場合の問題点として「メンテナンス工数の増加」を回答。メンターから「エラーレスポンス形式の一貫性が崩れるリスク」を補足
5. `200`と`201`の使い分けについて「同期/非同期の違い」という誤った基準で回答していたが、正しくは「新しいリソースが作られたかどうか」が基準であり、非同期処理を表すなら`202 Accepted`が本来使うべきコードであることを訂正・説明した

**つまずいた点**:
- DTOという用語自体を知らなかった（前提知識のため直接説明）
- 検証を分ける理由、ステータスコードの使い分けについて、表面的な理由（レスポンス速度、同期/非同期）で考えてしまう傾向があった。具体例・思考実験を通じて本質的な理由に気づけるよう誘導した
